Middleware for the Internet of Things
The principle of OneM2M is to offer a global standard for machine to machine and the Internet of Things. The objective of this association is to simplify the implementation of complex IoT networks integrating many devices. It is an open standard that offers a fairly generic and horizontal vision of connected systems. This responds to the need for a common service layer in the machine-to-machine domain, regardless of hardware or software. Thus with its standardized interface, OneM2M makes it possible to disregard the underlying protocols. This common layer tends to integrate different applications requiring for example their own communication protocols. This standard can be applied to a wide variety of sectors ranging from connected homes, e-health, intelligent transport and industrial automation.
OM2M is the Eclipse software dedicated to the deployment and implementation of the OneM2M standard. It is open-source software originally developed by LAAS-CNRS. At the beginning of the labs, we learned how to configure and launch the OM2M platform and how to manage the resource tree with the OM2M interface.During the labs, we were able to interact with objects and visualize the data sent by them. We used a java application launched on the user's computer to simulate lamps. You will find the ressource tree associated to the lamps in the following figure.
To manipulate resources I used a REST API. And I used Postman to easily send requests.
For the final project, we used Node-Red. It is an open-source visual editor from IBM for the Internet of Things. It makes it easy to link data sources to processing components, local or remote, and to create value chains with a few clicks. When the Node-Red program is deployed the operations are translated into http requests and sent to the MN-CSE to be translated into the technology of the device.
In the final project, we have created an application which turns lamp 1 OFF, when the lamp 0 is ON and the brightness is above a certain threshold (here 200 lux). When the brightness is below this threshold, the lamp 1 is turned ON. You will find our Node-Red wireframe in the following figure.
You will find further information on the course and labs in the following report.
Service Architecture
In this course, I studied SOA and ROA concepts. Nowadays, those concepts are essential in software engineering and very useful to build highly scalable, complex and modular applications. Service-oriented architecture is here to provide solutions to application integration problems. In order to expose the different functionalities of one or more services, the SOA encapsulates the applications in the form of software bricks called services. Unlike SOA (oriented at services), ROA (oriented at resources) will do more of its processing internally.
During the lab sessions I was able to put in practice all the knowledge acquired during the course. I created SOAP web services from a java class, following the bottom-up and top-down approach. I also implemented a client for the SOAP web service. An other lab session consisted in the creation, deployment and testing of a BPEL procedure orchestrating three web services. I also created a Rest web service and a client of this same service where the client would consume and produce data in JSON or XML format. In the final lab I used the Spring Boot framework to create three micro-services communicating and collaborating together.
In my final project, I deployed a RESTful architecture for a running competition. With Arnaud Guiller, we created the micro-services architecture in pairs and implemented two micro-services individually. We used the Maven management tool and the Spring Boot framework to create our numerous micro-services and we tested our implementation with Postman. You will find more details on this project in the report below.
Adaptability: Cloud and autonomous management
Cloud computing delivers computing services over the Internet to offer faster innovation, flexible resources, and economies of scale. It is harnessing the power of remote servers. Resources are pooled among several clients, which allows them to access “unlimited” resources remotely, without having to modify their local architecture. The cloud relies on machine virtualization and remote servers that adapt to the computing power needed. It is a huge step towards scalability and automatic adaptation of the load. There are several levels of cloud services, IaaS (the client manages from the OS of a VM), PaaS (the client controls a development environment), SaaS (the client controls only the data).
During the first lab, I discovered different virtualization techniques and how to use them. I configured a local virtual machine with VirtualBox. The general purpose of this lab was to enhance our knowledge of virtualization techniques, concepts and technologies.
We set up a Linux virtual machine on our computer using VirtualBox, and we tested hypotheses on the network connectivity. We also got to configure and manage containers.
In the final lab, I got a good understanding of the service-based application lifecycle management in cloud computing context. I studied the several provisioning process phases and investigated the way they should be implemented when provisioning. I discovered several “PaaS” cloud platforms and deployed a web applications.
You will find a detailed report of the course and labs in the following document.
Protocols for connected objects
Throughout this course we studied various technologies used in the internet of things. More and more present in our daily lives, connected objects help us to better manage our home, travels or health and facilitate digital transformation of companies. Technologies vary depending on the type of devices. To name only a few of the famous technologies, we have GSM, Wifi, ZigBee, Bluetooth. However, the world of IoT faces many challenges such as energy efficiency, security or deployment.
I studied the BLE technology, formerly known as Bluetooth Smart. This improvement of the standard Bluetooth appeared with version 4.0. It provides a service with greatly reduced energy consumption and cost, while maintaining the same coverage. It only works with recent devices because it is not backward compatible. The energy saving of this protocol comes largely from its possibility of being disconnected, that is to say that the devices are aware of each other and only connects when necessary, and as quickly as possible.
You will find further information in the following reports (synthesis of the course and study of the BLE technology).

Innovative Project
Throughout this semester I was involved in an innovative project alongside Franck Bourzat, Krishna Sujan Inuganti, Yuheng Jin, Ismail Touzani and Théo Zanchi. Our project aims to give people the opportunity to access and know information about the quality of their surrounding environment. This goal was achieved by building a device and a platform able to monitor pollution parameters at a specific location.
Our project is called "Social Network for Pollution" and our ambition is to distribute our solution to a large number of person. By displaying air quality measures on a map we want to create a social network around pollution monitoring. Solution: Our solution is composed of 4 different parts:
  • Device: A platform (LoPy4 board) is interconnected with the GPS receiver, some gas sensors: CO, O3, SO2, NO2 and more common sensors like temperature, humidity and pressure.
  • Transport: The platform sends the measures via the LoRa protocol to a worldwide LoRaWAN shared network.
  • Computation and storage: It is composed of an OM2M server which implements the TTN IPE and runs along a NodeJs server.
  • Visualisation: It is a mobile application. It fetches the data from the NodeJs server and displays them as a map.
You will find the general architecture of the project in the following figure.

Figure 5 - General architecture of the project

Device: We decided to monitor pollution by collecting data on the following parameters: Temperature, Pressure, Fine particles, Carbon monoxide, Ozone, Nitrogen dioxide and Sulphur dioxide. You will find the acquisition architecture (both high level and low level) in the following figures.

Figure 6 - High level design, Low level design

Transport: We needed to find a low energy protocol for our solution. LoRaWAN was a great choice. This MAC protocol allows long range wireless communication for battery powered devices. It covers both the second and third layer of the OSI model, which ease the creation of multi-devices networks. The Thing Network (TTN) is the biggest open shared LoRaWAN network and it was a great solution to implement the air quality monitoring platform. You will find a typical architecture of LoRaWAN network in the following figure.

Figure 7 - LoRaWAN architecture

Computation: We used an OM2M server to store and organize the data in our project. This open source software is the Eclipse implementation of the OneM2M standard. This standard has a transversal position that gives it a high degree of interoperability, adaptability and scalability with discovery and integration mechanisms. You will find the network architecture of the project in the following figure.

Figure 8 - Server architecture of the project

We chose to implement a noSQL MongoDB database for the user database because of the lightweight, simplicity and scalability. We implemented a REST API to make the transition between the OM2M server and the user applications. Visualisation: We chose React-Native to develop our mobil application. Unlike other crossplatform frameworks, React-Native provides native-like application performances and has numerous libraries and APIs.

Hackathon
In december, I participated in a 48 hours hackathon organised by Sierra Wireless and INSA Toulouse, on the future of connected mobility applied to IoT applications and services.
The project we developped is called "SafeDrive". It is a car accident detector combining an IoT device and a mobil application.
During the hackathon we were able to realise a prototype that could detect car accidents and automatically alert emergency services. The prototype was based on the MangOH card. We were able to use the built-in accelerometer to detect the collisions. When an accident was detected, we would send an SMS via a SIM card inserted in the MangOH. The SMS would contain various informations about:
  • the location (GPS coordinates and a google maps link)
  • the time of the accident
  • the driver's medical information (blood group, body size ...)
We printed a model of the car at the Fablab, in laser-cut wood. We also made an interactive mockup of the user mobile application using Adobe XD.
I mainly worked on the mobile application prototype, the business plan and the final presentation.
